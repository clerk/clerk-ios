# Shared session identity synchronization

Shared-session synchronization treats the device token and Client as one identity value. Each application owns one discoverable Keychain slot and is the only writer allowed to update or delete that slot. Applications enumerate compatible peer slots, reduce their events to one winner, persist the complete winning identity atomically in app-local Keychain storage, and then apply it to memory.

## Identity transition boundary

`ClerkIdentityController` is the single internal boundary for complete authentication identity transitions. Request preparation, network responses, Watch payloads, framework-driven device-token replacement, cache hydration, destructive cleanup, reconfiguration, and persisted Client-ID lookup enter through this controller. It owns response ordering, device-token response fences, the app-local identity operation queue, and the choice among legacy persistence, adopted atomic-local persistence, and active shared-session transport. `Clerk` memory is updated only from an authorized controller transition while atomic identity infrastructure is active.

The controller does not absorb transport-specific protocols. `SharedSessionSyncCoordinator` continues to own owner slots, event generations, reduction, notifications, pending shared publication recovery, and peer reconciliation. Watch synchronization continues to own Watch versions, fingerprints, and transport metadata. Atomic-identity and shared-slot Security work remains behind the existing actor-isolated I/O types rather than moving onto the main actor.

Keychain enumeration and mutation are serialized off the main actor. One coordinator-owned operation chain handles local publications, durable-pending recovery, peer reconciliation, winner replication, owner-slot deletion, request identity capture, and network-response resolution. Raw response commands enter that chain before consulting the current token or Client, so a queued token-only response cannot retain identity state that an earlier queued response subsequently replaces. Every Clerk request requires a successful initial reconciliation and retries that reconciliation after a transient failure. The request then captures its token, Client ID, response-generation context, and shared frontier as one queue-ordered snapshot. Network requests retain that first prepared snapshot across retries only while a freshly prepared attempt still observes the same complete identity context. A token, Client ID, response-generation, or shared-frontier change cancels the retry before transmission. The captured Clerk token is also stored as typed request metadata before custom middleware runs; response processing never mistakes a custom proxy/signing `Authorization` value for Clerk's device token. Custom request middleware runs after the retained Clerk context is restored so it observes the headers that will be transmitted.

An observed frontier is not usable for request preparation until its winning identity has been durably accepted and applied. A failed reconciliation leaves request capture and later local publication behind a mandatory reconciliation retry, preventing a newer generation from being paired with stale credentials. App-local request capture after shared transport is disabled likewise reserves a position in the same revisioned identity queue as Watch and network transitions.

## Event order

Compatible, valid events use this lexicographic total order:

1. Higher logical generation.
2. At equal generation, an event with a server date over an event without one.
3. At equal generation with dates, the later server date.
4. Lexicographically greater origin owner identifier.
5. Lexicographically greater event UUID string.

The reducer deduplicates identical replicated events, excludes every payload involved in a conflicting reuse of one event ID, and separately reports the greatest valid observed generation. It does not use Client `updatedAt` or give sign-out special precedence. A response normally publishes only when its captured frontier equals the coordinator's accepted frontier. Newer responses that were prepared from the same frontier may extend a process-local network-response lineage in response-sequence/server-date order, with Client `updatedAt` used only to distinguish responses that share the same server date. A fresh generation is allocated when the serialized operation runs. Any Watch, local, peer, token-changing, or conflicting winner breaks that lineage; same-owner events are not an exception.

## Bounded pending publication

The app-local atomic record may contain the accepted identity and at most one immutable pending publication event. The bounded pending intent is an approved part of the persistence model.

A local transition is first staged as that exact pending event. The same event ID, owner, generation, payload, and server date are then written to the app's owner slot. After reducing all compatible slots, the selected winner is committed as the accepted local identity and the matching pending intent is cleared. A retry reuses the exact event and never rebases it. New publication work is serialized behind pending recovery.

The record does not claim a transaction across two independent Keychain items: process termination can still occur between the app-local stage and owner-slot write, or after the owner-slot write and before the accepted local commit. Restart recovery resolves both windows from the immutable pending event and observed slots.

## Response and payload semantics

Successful frontend mutations can return an operation object in `response` and the authoritative Client snapshot in the sibling `client` field. Errors can carry the snapshot in `meta.client`. A removed Session plus a signed-out Client with an empty `sessions` array is therefore a complete Client update, not an identity clear. Null piggyback fields mean no Client update. A canonical `/v1/client` response with `response: null` and `client: null` also means no usable Client update, including the backend's database-maintenance response. The native `DELETE /v1/client` contract is a separate explicit teardown: its `Authorization: Bearer ` response header clears the device token and complete local identity even if `response` contains the deleted Client snapshot.

One response checkpoint owns both the Authorization-token transition and the Client transition. Request sequence, server date, and device-token generation are evaluated once before either half can change, so a rejected Client response cannot still rotate the token. Because response work is serialized, its ordering watermark is committed only after the complete identity becomes durable and selected; a failed local stage therefore cannot consume the sequence needed by an exact retry. Canonical refresh decoding does not reinterpret a missing Client as a second clear after middleware has classified the raw payload. After shared-session adoption, the app-local token is hydrated into memory for request preparation, while atomic response persistence is serialized through off-main identity I/O before memory changes.

Framework-driven device-token replacement follows the same complete-identity rule even before adoption. The new token is persisted, the previous Client and ordering date are cleared while independent Watch publication is suppressed, and one coherent identity-change event is emitted before canonical refresh. Watch never observes a new token paired with the previous Client.

Watch payloads are validated as complete token/Client transitions before publication. A Client snapshot must carry its paired nonempty token, a changed token without a Client clears the previous Client and requests a canonical refresh, and each Watch arrival synchronously reserves its place in the shared coordinator's serialized operation chain. After shared transport is disabled, Watch and network transitions continue through the same revisioned app-local identity queue. Watch version metadata is app-local and read failures or corruption reject the identity update. One outgoing sync resolves pending metadata once and constructs the payload from that single snapshot. Before applying an incoming identity, the implementation durably stages its version, state, and payload fingerprint; it promotes the version and fingerprint only when the submitted identity becomes the selected shared winner. A candidate superseded by a concurrent peer winner discards its matching pending metadata so an exact retransmission can be evaluated again. An exact retransmission may complete a pending transition, an exact replay of an accepted transition is a no-op, and conflicting reuse of a pending or accepted version is rejected. Refresh and publication task cleanup is scoped by task identity so cancellation before a queued operation begins cannot retain stale task state or clear replacement work.

## Compatibility and cleanup

Malformed, foreign-instance, owner/account-mismatched, and unsupported future-schema slots are excluded. A minimal schema-version header is decoded before current fields, so a future-schema own slot is preserved even if that schema renames or removes today's header fields. The current SDK neither overwrites nor deletes it, while still adopting compatible peer winners locally. Keychain access groups are normalized once for topology comparison, Security queries, and notification names so whitespace cannot split one logical topology.

Adoption runs only after the previous runtime's writers are stopped and drained. Existing atomic identity wins, followed by a nonempty token from the configured app-local legacy source, the previous bundle-local source, and finally legacy shared storage. Legacy token, Client, and date fields were independently written and have no common revision, so adoption never promotes their apparent tuple to an atomic identity. It adopts only the single-item token as provisional cleared state; the canonical Client refresh establishes the first token/Client snapshot. Environment may fall through every validated source. Magic-link and App Attest state migrate only from app-attributed storage and are not imported from ambiguous sibling-shared storage.

Destructive local cleanup immediately fences prepared responses and clears the live atomic identity. It invalidates older identity revisions, freezes cache persistence, drains coordinator and cache writers, deletes only this application's owner slot, scrubs reusable credentials from the adopted legacy shared store, and performs final app-local deletion before releasing the reconciliation barrier. Every suspended coordinator operation revalidates the configuration, clear state, and operation revision before another shared write. If owner-slot withdrawal fails, the clear reports the failure and deliberately leaves shared reconciliation and request capture barriered until a later clear retry succeeds. Overlapping clear calls join the same transaction so persistence cannot resume between them. A credential-free Watch clear tombstone advances both Watch versions, replaces old fingerprints, and remains durable through explicit clears and strict reconfiguration so stale peer payloads or legacy metadata fallback cannot reseed the identity. The synchronous `clearAllKeychainItems()` API preserves source compatibility, performs its immediate best-effort pass, and schedules the serialized cleanup. The throwing `clearAllKeychainItemsAndWait()` API waits for the complete writer barrier, app-local deletion, and owner-slot withdrawal and reports any boundary that could not be cleared. Reconfiguration performs credential adoption only after draining the old runtime and only when the normalized publishable key permits identity preservation. Destructive or cross-key transitions clear the destination and write the adoption marker without importing legacy credentials. The old owner slot remains available for rollback through every other fallible step and is withdrawn as the final required boundary before the new runtime is committed. The owner slot is preserved only when the key and complete slot topology remain unchanged; peers are never deleted.

If reconfiguration begins while a public Keychain clear is active, the clear transaction completes before the old runtime can be invalidated or any same-key state-preservation decision is made. Watch ordering versions use checked advancement and fail explicitly if the representable version space is exhausted.
